# Software Defined Vehicle / Software Defined Robot

## 소프트웨어가 가치를 결정하는 시대

2012년 Tesla Model S가 처음으로 주행 중 OTA(Over-The-Air) 업데이트를 시작했을 때, 자동차 산업의 패러다임이 바뀌었다. 자동차를 판매한 후에도 소프트웨어로 성능을 개선하고 기능을 추가할 수 있다는 것—이것이 SDV(Software Defined Vehicle)의 핵심이다. 구매자는 처음 구매한 하드웨어 위에서 수년에 걸쳐 새로운 기능을 받는다.

의료 로봇에서도 동일한 전환이 일어나고 있다. SDR(Software Defined Robot)이라는 표현이 아직 산업 표준 용어는 아니지만, 그 방향성은 분명하다. FDA가 AI/ML 기반 의료기기를 위한 "Predetermined Change Control Plan"을 인정하기 시작한 것이 그 증거다—즉, 판매 후 소프트웨어 업데이트로 의료기기의 기능을 개선하는 것을 규제 프레임워크가 허용하기 시작했다.

---

## SDV 아키텍처의 전제: 하드웨어 추상화

SDV가 가능하려면 소프트웨어가 특정 하드웨어에 묶여 있으면 안 된다. 이것이 하드웨어 추상화 계층(HAL)이 중요한 이유다.

```
SDV/SDR 소프트웨어 스택 (아래서 위로):

┌──────────────────────────────────────────────────────────────────┐
│  비즈니스/서비스 레이어                                           │
│  Feature on Demand, 원격 진단, 데이터 수익화, 구독 모델          │
├──────────────────────────────────────────────────────────────────┤
│  애플리케이션 레이어                                              │
│  수술 보조 AI, 관절 제어 알고리즘, HMI, 원격 모니터링            │
├──────────────────────────────────────────────────────────────────┤
│  미들웨어 레이어                                                  │
│  ROS 2 + DDS, AUTOSAR Adaptive (ara::com), SOME/IP              │
├──────────────────────────────────────────────────────────────────┤
│  가상화 레이어                                                    │
│  Hypervisor (QNX / ACRN / Xen), Container Runtime               │
├──────────────────────────────────────────────────────────────────┤
│  OS 레이어                                                       │
│  RTOS (QNX, Zephyr) + Linux (Yocto 기반)                        │
├──────────────────────────────────────────────────────────────────┤
│  하드웨어 추상화 레이어 (HAL / BSP)                               │
│  Ethernet TSN 드라이버, CAN 드라이버, GPU API (CUDA/OpenCL)      │
├──────────────────────────────────────────────────────────────────┤
│  하드웨어                                                        │
│  HPC SoC, Zone ECU MCU, TSN 스위치, 센서/액추에이터             │
└──────────────────────────────────────────────────────────────────┘
```

HAL이 제대로 설계되면 관절 제어 알고리즘이 특정 모터 드라이버 칩에 의존하지 않는다. 모터 드라이버가 교체되어도 HAL만 업데이트하면 상위 소프트웨어는 그대로 동작한다. 이것이 하드웨어 설계 변경과 소프트웨어 개발을 독립적으로 진행할 수 있게 해준다.

## OTA: 업데이트가 의료기기에서 의미하는 것

일반 소비 제품의 OTA 업데이트가 실패하면 재부팅하면 된다. 의료 로봇의 OTA 업데이트가 실패하면 다음 날 예약된 수술이 취소될 수 있다. 이 차이가 의료 로봇 OTA의 모든 설계 결정을 지배한다.

### A/B 파티션: 안전한 업데이트의 기반

```
스토리지 레이아웃 (eMMC 128GB 예시):

┌─────────────────────────────────────────────────────────────────┐
│ 파티션 레이아웃                                                  │
│                                                                 │
│  [Boot A]   [Boot B]   [Rootfs A]        [Rootfs B]    [Data]  │
│  256MB      256MB      8GB (현재 실행)    8GB (업데이트) 112GB  │
│    ↑                     ↑                                      │
│  현재 부트로더          v2.4.1 실행 중                           │
│                                                                 │
│  OTA 진행 중:                                                   │
│    → Rootfs B에 v2.5.0 기록 (서비스 중단 없음)                  │
│    → Boot B 업데이트                                            │
│    → Health Check 후 A→B 전환                                   │
│    → 실패 시 A로 자동 롤백                                       │
└─────────────────────────────────────────────────────────────────┘

핵심: 업데이트 중 현재 파티션(A)은 완전히 정상 동작
      새 버전(B) 검증 완료 전까지 절대 A를 덮어쓰지 않음
```

### UPTANE: 자동차/의료용 OTA 보안 프레임워크

단순한 HTTPS 다운로드로는 충분하지 않다. 공격자가 서버를 해킹해 악성 이미지를 배포하거나, 구버전 취약점이 있는 이미지로 다운그레이드하거나, 특정 장치를 타깃으로 공격할 수 있다. UPTANE은 이 모든 공격 벡터를 고려한 프레임워크다.

```
UPTANE 구조:

[Image Repository]              [Director Repository]
  모든 공개 이미지 저장            특정 차량/로봇에 무엇을 설치할지 결정
  Timestamp 서명                  ECU별 개별 지시(Directive) 서명
  Snapshot 메타데이터             실시간 업데이트 상태 추적
        │                                    │
        └──────────────┬─────────────────────┘
                       │ HTTPS (TLS 1.3)
                ┌──────▼───────┐
                │  OTA Client  │
                │  (HPC 탑재)  │
                └──────┬───────┘
                       │ 검증 체계
                       │ 1. Director 서명 검증 (이 장치에 맞는 이미지인가?)
                       │ 2. Image Repo 서명 검증 (이미지 위변조 없는가?)
                       │ 3. Timestamp 검증 (리플레이 공격인가?)
                       │ 4. 버전 번호 검증 (다운그레이드인가?)
                       │ 5. ECU별 메타데이터 검증 (올바른 ECU 대상인가?)
                ┌──────▼───────┐
                │   설치 실행   │
                └──────────────┘
```

**의료기기 특수 요구사항:** FDA 2023 사이버보안 가이드라인은 SBOM(Software Bill of Materials)을 OTA 업데이트 패키지에 포함할 것을 권고한다. 각 업데이트에 어떤 오픈소스 라이브러리가 포함되어 있고, 알려진 CVE 취약점은 무엇인지 추적 가능해야 한다.

### 델타 OTA와 전체 이미지 OTA

```
전략 비교:

전체 이미지 OTA:
  패키지 크기: Rootfs 전체 (1~4GB)
  전송 시간: 병원 1Gbps 인터넷으로 30~60분
  장점: 단순, 신뢰성 높음
  단점: 대용량, 병원 네트워크 부담
  적합: 주요 버전 업그레이드 (연 1~2회)

델타 OTA:
  패키지 크기: 변경된 파일만 (10~100MB)
  전송 시간: 수분
  장점: 소용량, 빠른 배포
  단점: 패치 생성 복잡, 무결성 검증 필요
  적합: 버그 패치, 소규모 기능 개선 (월 1~2회)
  도구: bsdiff, zstd-delta, OSTree
```

### IEC 62304와 OTA 업데이트

OTA 업데이트가 의료기기 소프트웨어를 변경하는 것은 IEC 62304의 범위다. 매 OTA 패치는:

1. **변경 영향 분석(Change Impact Analysis):** 변경된 소프트웨어 유닛이 어떤 기능에 영향을 주는지 추적
2. **회귀 테스트:** 변경된 유닛과 그 의존 유닛의 테스트 재수행
3. **소프트웨어 안전 등급 재평가:** Class C 기능에 영향을 주면 전체 재검증 필요
4. **출시 전 검토:** 의료기기 품질 시스템(QMS)의 소프트웨어 출시 절차

이 때문에 의료 로봇의 OTA 업데이트 주기는 소비자 전자기기보다 훨씬 길다. Tesla는 주간 단위로 업데이트를 배포하지만, 의료 로봇은 각 업데이트가 규제 검토를 거치기 때문에 분기 또는 반기 단위가 현실적이다.

## CI/CD 파이프라인: 의료기기 품질에 맞춘 자동화

```
소스 코드 변경 (Git Push to feature branch)
        │
        ▼
┌──────────────────────────────────────────────────────────────────┐
│  CI 파이프라인 (자동 실행, ~30분)                                 │
│                                                                  │
│  Step 1: 정적 분석                                               │
│    MISRA-C:2012 검사 (안전 코드)                                  │
│    Clang-Tidy / cppcheck (일반 코드)                             │
│    SAST 보안 취약점 스캔 (Coverity / SonarQube)                  │
│                                                                  │
│  Step 2: 단위 테스트                                             │
│    Google Test (C++) / pytest (Python)                           │
│    코드 커버리지 측정 (IEC 62304 Class C: MC/DC 커버리지 필요)   │
│    경계 조건 테스트 (최대값, 최소값, 오버플로우)                   │
│                                                                  │
│  Step 3: 크로스 빌드                                             │
│    Yocto BitBake: x86-64 → ARM64 크로스 컴파일                  │
│    빌드 재현성 검증 (동일 소스 → 동일 바이너리 hash)             │
│                                                                  │
│  Step 4: SIL (Software-in-the-Loop) 테스트                      │
│    로봇 동역학 시뮬레이터 (MuJoCo / ROS 2 + Gazebo)             │
│    제어 알고리즘 기능 검증                                        │
│    네트워크 지연 주입 테스트 (tc netem)                           │
└──────────────────────────────────────────────────────────────────┘
        │ (모든 검사 통과)
        ▼
┌──────────────────────────────────────────────────────────────────┐
│  CD 파이프라인 (승인 후 실행, ~2시간)                             │
│                                                                  │
│  Step 5: HIL (Hardware-in-the-Loop) 테스트                      │
│    실제 Zone ECU + 모터 드라이버 테스트 벤치                     │
│    실제 네트워크에서 TSN 타이밍 검증                              │
│    Fault injection: 링크 단선, PTP 실패, 패킷 손실               │
│                                                                  │
│  Step 6: OTA 패키지 생성 및 서명                                 │
│    UPTANE Director + Image 서명 (코드 서명 키, HSM에 저장)       │
│    SBOM 생성 (syft, grype)                                       │
│    패키지 무결성 해시 (SHA-256)                                   │
│                                                                  │
│  Step 7: 스테이징 배포 (랩 환경 로봇)                             │
│    72시간 내구성 테스트                                           │
│    성능 회귀 테스트 (제어 지연, 지터 측정)                        │
│                                                                  │
│  Step 8: 점진적 롤아웃 (프로덕션)                                │
│    1대 → 5대 → 20% → 100% 순차 배포                             │
│    자동 모니터링 + 이상 감지 시 자동 롤백                         │
└──────────────────────────────────────────────────────────────────┘
```

**MC/DC(Modified Condition/Decision Coverage):** IEC 62304 Class C는 MC/DC 코드 커버리지를 요구한다. 이것은 단순한 라인 커버리지(모든 코드가 실행됨)보다 훨씬 강력한 기준으로, 각 조건이 독립적으로 결과에 영향을 미침을 증명해야 한다. `if (a && b)` 문에서 MC/DC를 달성하려면 a만 변경해서 결과가 달라지는 테스트와 b만 변경해서 결과가 달라지는 테스트가 각각 필요하다.

## Feature on Demand: 의료 로봇의 비즈니스 모델 혁신

SDV/SDR 아키텍처는 새로운 비즈니스 모델을 가능하게 한다.

```
기본 패키지 (초기 구매):
  ├── 복강경 수술 기본 모드
  ├── 3K 내시경 영상 (30fps)
  └── 기본 안전 기능

선택적 활성화 기능 (라이선스 키로 잠금 해제):
  ├── 4K HDR 내시경 영상 (60fps)  ← 하드웨어는 이미 존재
  ├── AI 조직 인식 보조 기능
  ├── 햅틱 피드백 고급 모드
  ├── 원격 수술 지원 모드 (원격 의사 보조)
  └── 특수 수술 도구 지원 패키지

구독 기반 서비스:
  ├── 원격 진단 및 예방 유지보수 알림
  ├── AI 모델 지속 업데이트 (분기별)
  └── 수술 데이터 분석 대시보드
```

**규제적 고려사항:** Feature on Demand 모델에서 새로운 기능을 활성화하는 것이 새로운 의료기기 허가가 필요한지에 대해 FDA와 사전 논의가 필요하다. FDA의 "Predetermined Change Control Plan"(2021 guidance)이 이 문제를 다루기 시작했다—사전에 변경 범위를 명시하고 승인받으면 각 업데이트마다 별도 허가 없이 배포할 수 있다.

## Yocto: 내장 Linux의 표준 빌드 시스템

의료 로봇 HPC의 Linux 이미지 빌드에 Yocto Project가 널리 사용된다.

```bash
# Yocto 기본 구조
poky/                    # Yocto 기본 레이어
  meta/                  # 핵심 레시피
  meta-poky/             # Poky 설정

meta-medical-robot/      # 프로젝트 전용 레이어
  conf/
    layer.conf
  recipes-kernel/
    linux-medical/        # 커널 설정 (RT-Preempt, TSN 모듈 포함)
  recipes-connectivity/
    linuxptp/             # PTP 스택
    iproute2-tc/          # tc taprio, tc cbs 포함
  recipes-robotics/
    ros2-humble/          # ROS 2 cross-compile 설정
    medical-robot-app/    # 병원 소프트웨어

# 빌드 명령
bitbake core-image-medical-robot
# → ARM64 rootfs 이미지, 커널, DTB 생성
# → SBOM 자동 생성 (meta-spdx 레이어 사용)
```

Yocto의 중요한 장점은 **빌드 재현성(reproducible builds)**이다. 동일한 레시피로 언제 빌드해도 동일한 바이너리가 나와야 IEC 62304의 형상 관리 요구사항을 만족한다. 이를 검증하기 위해 두 번 빌드한 이미지의 SHA-256 해시를 비교하는 것이 좋은 실천이다.
