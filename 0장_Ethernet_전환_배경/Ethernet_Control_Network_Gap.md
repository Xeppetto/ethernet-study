# 0.3 하지만 Ethernet은 제어 네트워크가 아니다 (Ethernet's Fundamental Gap in Control)

## 개요

0.2에서 Ethernet의 강점을 살펴봤습니다. 그렇다면 "Ethernet으로 로봇을 제어하면 되지 않나?"라는 질문이 자연스럽게 나옵니다. **결론부터 말하면, 일반 Ethernet은 안전 임계적 실시간 제어에 적합하지 않습니다.** 그 이유를 이 문서에서 정확히 이해하는 것이 TSN(3장)의 필요성을 납득하는 가장 빠른 방법입니다.

---

## 1. Ethernet의 설계 철학: Best-Effort

### 1.1 Best-Effort란?

```
Best-Effort 네트워크의 정의:
  "네트워크는 최선을 다해 데이터를 전달하려 하지만,
   전달 시간, 전달 순서, 전달 성공 여부를 보장하지 않는다."

인터넷의 설계 철학 (1970년대):
  → 핵전쟁에서도 살아남는 강인한 네트워크
  → 어떤 경로로든 목적지에 도달
  → 경로 일부가 파괴되어도 우회하여 전달
  → "반드시 도달하되, 언제 도달할지는 보장 안 함"

이 철학의 결과:
  ✓ 매우 강인한 라우팅 (인터넷의 핵심 강점)
  ✓ 다양한 물리 매체 지원 (구리, 광섬유, 위성, 무선)
  ✓ 네트워크 상태에 따른 유연한 경로 선택
  ✗ 전달 시간 보장 없음 → 실시간 제어에 치명적
  ✗ 전달 순서 보장 없음 → 제어 명령 순서 뒤바뀔 수 있음
  ✗ 전달 성공 보장 없음 → 패킷 손실 → 제어 루프 파손
```

### 1.2 Ethernet 스위치의 "최선 노력" 동작

```
패킷 처리 흐름 (Best-Effort):

입력 포트 → 수신 버퍼 → 스위칭 패브릭 → 출력 버퍼 → 출력 포트

출력 버퍼가 포화 상태일 때:
  선택 1: 새 패킷 폐기 (Tail Drop)
  선택 2: 오래된 패킷 폐기 (RED: Random Early Detection)
  → 어떤 선택이든 "패킷 폐기"는 발생

QoS(우선순위 큐) 설정 시:
  높은 우선순위 큐: 낮은 우선순위보다 먼저 처리
  하지만: 높은 우선순위 패킷이 이미 전송 시작된 낮은 우선순위 프레임에 의해 지연
  → 완전한 결정론적 보장 아님

결론: 표준 Ethernet은 "더 낮은 지연을 노력"할 뿐, "지연을 보장"하지는 않음
```

## 2. 비결정성의 원인 분석

### 2.1 원인 1: 큐잉 지연 (Queuing Delay)

```
시나리오: 1Gbps 스위치, 출력 포트에 3개 입력 동시 도달

포트 1: 1518 Byte 영상 데이터   ──────────────────────►
포트 2: 1518 Byte 배경 데이터   ──────────────────────►  출력 포트
포트 3:   64 Byte 긴급 제어 명령 ──────────────────────►  (1Gbps)

이상적 기대: 긴급 제어 명령이 즉시 전달
실제 현실:
  출력 큐에서 순서대로 처리 (FIFO):
  1518 Byte 전송: 12.2 µs
  1518 Byte 전송: 12.2 µs
  64 Byte 전송:    0.7 µs
  → 긴급 명령은 24.4 µs 대기 후 전송!

QoS(우선순위 큐) 설정 시:
  긴급 명령(PCP=5)이 먼저 처리됨
  하지만: 이미 전송 시작된 1518 Byte 프레임은 막을 수 없음
  → 최대 12.2 µs (1518 Byte @ 1Gbps) 불가피한 HOL(Head-of-Line) 지연

총 가능한 큐잉 지연 범위:
  최솟값:  0 µs (경쟁 없을 때)
  최댓값:  8,000 µs = 8 ms (1MB 버퍼 @ 1Gbps)
  → 0~8,000 µs 범위의 비결정적 지연
```

### 2.2 원인 2: Broadcast Storm과 브로드캐스트 부하

```
네트워크 초기화 및 토폴로지 변경 시:

ARP Request 브로드캐스트:
  새 노드 접속 → "192.168.10.1을 아는 사람?" → 전체 브로드캐스트
  → 동일 서브넷 모든 노드 인터럽트 발생
  → 로봇 제어 ECU가 ARP 처리 중 제어 루프 지연 가능

STP(Spanning Tree) 재수렴:
  링크 장애 감지 → RSTP Topology Change Notification → 브로드캐스트
  → 모든 스위치 MAC 테이블 플러시 → 일시적 Flooding
  → 수 초간 모든 유니캐스트 트래픽이 브로드캐스트처럼 동작

브로드캐스트 스톰 시:
  루프 미감지 스위치에서 브로드캐스트 무한 순환
  → 수 초 내 네트워크 완전 마비
  → 안전 제어 시스템 통신 두절

대응:
  VLAN으로 브로드캐스트 도메인 분리 (부분 완화)
  TSN 환경: 루프 없는 토폴로지 고정 + FRER으로 이중화 (근본 해결)
```

### 2.3 원인 3: TCP 재전송 메커니즘의 위험성

```
TCP와 실시간 제어의 근본적 불일치:

TCP 재전송 시나리오:
  제어기 → [제어 명령 1] → 로봇 (T=0ms)
  제어기 → [제어 명령 2] → 로봇 (T=1ms)
  제어기 → [제어 명령 3] → 로봇 (T=2ms) ← 패킷 손실!

  TCP 재전송 타이머 (RTO, Retransmission Timeout):
    최솟값: 200ms (Linux 기본값)
    → T=202ms에 재전송 시도
    → 로봇은 200ms 동안 마지막 수신 명령 유지 또는 대기
    → 200ms = 1kHz 제어 사이클의 200배!

  만약 의료 수술 로봇이 200ms 동안 제어 명령 없이 동작한다면?
    → 관절이 마지막 명령 방향으로 계속 이동
    → 0.5m/s 속도 관절: 200ms × 0.5m/s = 10cm 이동
    → 수술 중 10cm 오작동 → 환자 안전 위협

TCP 혼잡 제어의 추가 위험:
  CWND(혼잡 윈도우) 감소: 패킷 손실 감지 → 전송 속도 반으로 감소
  → 다음 수 사이클 동안 제어 명령 전달 속도 저하
  → 제어 루프 성능 불안정

결론: 안전 임계 제어 명령에 TCP 사용 금지
  → UDP 사용 + 애플리케이션 수준의 시퀀스 번호 + Watchdog 타이머
```

### 2.4 원인 4: 소프트웨어 스택 처리 지연

```
표준 Linux에서 UDP 패킷 수신 경로:

하드웨어 DMA → NIC 링 버퍼 → 하드웨어 인터럽트 →
소프트IRQ 스케줄 → 소켓 수신 버퍼 → 애플리케이션 recv() 호출

각 단계별 지연:
  DMA → NIC 버퍼:      수십 ns (하드웨어)
  하드웨어 인터럽트:    수십 ns ~ 수 µs (CPU 간섭 의존)
  소프트IRQ:            수 µs ~ 수십 µs (커널 부하 의존)
  OS 스케줄링:          0 ~ 수십 ms (다른 프로세스 실행 중이면 지연)
  recv() 반환:          수 µs (시스템 콜 오버헤드)

총 소프트웨어 스택 지연:
  표준 Linux (비RT): 수십 µs ~ 수 ms (매우 가변적)
  RT-Preempt Linux:  수 µs ~ 수십 µs (개선됨)
  RTOS:              수백 ns ~ 수 µs (결정론적)

→ 표준 Linux에서는 OS 스케줄링 자체가 Jitter의 주요 원인
→ 실시간 제어에는 RT-Preempt 패치 또는 RTOS 필수
```

## 3. Jitter: 제어 시스템의 침묵의 적

### 3.1 Jitter란?

```
정의: 연속적인 데이터 전달 사이의 지연 변동량

이상적 제어 (Jitter = 0):
  T=0ms:    명령 1 도착
  T=1ms:    명령 2 도착  (간격 정확히 1ms)
  T=2ms:    명령 3 도착  (간격 정확히 1ms)
  ...

현실적 Ethernet (Jitter 있음):
  T=0ms:     명령 1 도착
  T=0.8ms:   명령 2 도착  (간격 0.8ms, 0.2ms 빠름)
  T=2.3ms:   명령 3 도착  (간격 1.5ms, 0.5ms 늦음)
  T=2.9ms:   명령 4 도착  (간격 0.6ms, 0.4ms 빠름)
  ...

제어기 관점:
  명령 간격이 들쑥날쑥 → 실제 속도/가속도 계산 오류
  → 보간(Interpolation) 오류 → 경로 정밀도 저하
  → 힘 제어의 경우 순간적 과부하/미달 발생
```

### 3.2 Jitter가 제어 시스템에 미치는 영향

```
사례 1: 위치 제어 (Position Control)

PID 제어기 (이산 시간):
  u(k) = Kp × e(k) + Ki × Σe × T + Kd × (e(k) - e(k-1)) / T
  (T = 샘플링 주기, 이상적으로 1ms)

Jitter 발생 시 (T가 0.8ms~1.5ms로 변동):
  Kd × (e(k) - e(k-1)) / T 항:
    T=0.8ms 시: Kd × Δe / 0.0008 = Kd × 1250 × Δe  (계획의 125%)
    T=1.5ms 시: Kd × Δe / 0.0015 = Kd ×  667 × Δe  (계획의 66.7%)
  → 미분 항이 ±40% 변동 → 제어기 출력 불안정 → 진동 발생

사례 2: 힘 제어 (Force/Impedance Control)

임피던스 제어 (수술 로봇 조직 접촉):
  F_ref = M × x_ddot + B × x_dot + K × (x - x_eq)
  (M: 가상 질량, B: 가상 감쇠, K: 가상 강성)

Jitter 영향:
  x_dot = (x(k) - x(k-1)) / T  ← T 변동으로 오차 발생
  → 가상 감쇠력 계산 오류 → 실제 접촉력 변동
  → 수술 중 떨림(Tremor) 현상 발생 → 조직 손상 위험

사례 3: 다관절 동기 제어

6-DOF 로봇 팔 끝단 경로 제어:
  관절 1~6 모두 정확히 T=1ms에 새 위치 명령 받아야 함

Jitter 시:
  관절 1: T=0.9ms에 명령 수신
  관절 3: T=1.2ms에 명령 수신 (0.3ms 차이!)
  → 관절 3이 지연된 0.3ms 동안 관절 1은 이미 이동
  → 끝단이 계획 경로에서 이탈 → 정밀도 저하, 충돌 위험
```

### 3.3 허용 Jitter 수준

```
제어 시스템별 Jitter 허용 범위:

산업 로봇 (일반):
  허용 Jitter:  ±1ms 이내
  사이클 타임:  4~8ms (125~250Hz)
  → EtherCAT, PROFINET IRT로 구현 가능

협동 로봇 (Cobot, e.g. UR5, Franka):
  허용 Jitter:  ±100µs 이내
  사이클 타임:  2ms (500Hz)
  → EtherCAT 또는 TSN Ethernet 필요

수술 로봇 (의료):
  허용 Jitter:  ±10~50µs 이내 (요구사항에 따라 다름)
  사이클 타임:  1ms (1kHz) ~ 500µs (2kHz)
  → TSN Ethernet (802.1Qbv TAS)이 현실적 해결책

신경외과 로봇 (초정밀):
  허용 Jitter:  ±5µs 이내
  사이클 타임:  200µs ~ 500µs
  → TSN + RTOS + RT-Preempt Linux 조합

일반 Best-Effort Ethernet의 실측 Jitter:
  경량 부하 (< 10%): 수십 ~ 수백 µs
  중간 부하 (50%):   수백 µs ~ 1 ms
  고부하 (> 80%):    1 ms ~ 수십 ms
  → 대부분 제어 요구사항 초과
```

## 4. 실시간성의 한계: 정량적 분석

### 4.1 Ethernet 스위치 홉당 지연 분포

```
실측 데이터 (1Gbps 상용 스위치, 배경 트래픽 50%):

┌─────────────────────────────────────────────────────────────────┐
│ 지연 분포 (히스토그램)                                           │
│                                                                 │
│  빈도                                                           │
│   │                                                             │
│   │  █████                                                      │
│   │  ████████                                                   │
│   │  ██████████                                                 │
│   │  ████████████                                               │
│   │  ██████████████                                             │
│   │  ████████████████                         █                │
│   │  ██████████████████                    ████                │
│   └──────────────────────────────────────────────────────── 지연│
│      10µs  50µs  100µs  500µs  1ms  2ms  5ms  8ms             │
│                                                                 │
│  최솟값: ~12 µs (직렬화 지연만, 큐잉 없음)                      │
│  95th 퍼센타일: ~500 µs                                         │
│  99th 퍼센타일: ~2 ms                                           │
│  최댓값:  ~8 ms (버퍼 포화)                                     │
└─────────────────────────────────────────────────────────────────┘

TSN TAS 적용 후 (동일 환경):
  최솟값: ~14 µs
  95th 퍼센타일: ~16 µs  ← 거의 변화 없음!
  99th 퍼센타일: ~18 µs
  최댓값:  ~20 µs  ← Jitter ≈ ±3 µs 수준
```

### 4.2 홉 수 증가에 따른 지연 누적

```
다중 홉 네트워크에서 지연 누적 (1Gbps, 1518 Byte):

         Store-and-Forward 지연 모델
홉 수    최소 지연           최대 지연(Best-Effort)
 1      ~25 µs             ~8 ms
 2      ~50 µs             ~16 ms
 3      ~75 µs             ~24 ms
 5     ~125 µs             ~40 ms

수술 로봇 제어 요구:
  편도 지연: < 200 µs (왕복 < 400 µs)
  사이클 타임: 1ms

→ 2홉 이상 구성에서 Best-Effort Ethernet은 지연 요구사항 충족 불가
→ TSN 적용 시: 홉당 추가 지연 ~12 µs (직렬화) + 처리 ~2 µs
  5홉 TSN: ~(14 + 12×4) µs ≈ 62 µs → 200 µs 요구사항 충족 가능
```

### 4.3 표준 Linux에서의 소프트웨어 지연

```
cyclictest 측정 결과 비교 (1kHz, 1시간):

일반 Linux (커널 5.15, 비RT):
  최솟값:   5 µs
  평균:    35 µs
  최댓값: 8,500 µs = 8.5 ms  ← 커널 시스템 콜, GC 등에 의한 스파이크
  → 실시간 제어에 절대 부적합

RT-Preempt Linux (PREEMPT_RT):
  최솟값:   5 µs
  평균:    15 µs
  최댓값:  85 µs  ← 크게 개선됨
  → 소프트 실시간 요구사항 충족 가능

RTOS (QNX, VxWorks):
  최솟값:   1 µs
  평균:     3 µs
  최댓값:  20 µs
  → 하드 실시간 요구사항 충족 가능

TSN + RT-Preempt 조합 (권장):
  네트워크 지연: < 20 µs (TSN 보장)
  OS 처리 지연:  < 85 µs (RT-Preempt)
  총 End-to-End: < 105 µs → 200 µs 요구사항 충족
```

## 5. Best-Effort의 안전성 문제

### 5.1 제어 시스템 안전 요구사항과의 충돌

```
IEC 61508 (기능 안전) 핵심 요구:
  "안전 시스템은 알려진 최악의 조건(WCRT: Worst-Case Response Time)
   내에서 반드시 안전 기능을 수행해야 한다."

Best-Effort Ethernet으로 WCRT 계산 불가:
  WCRT = f(네트워크 부하, 다른 트래픽, 버퍼 크기, ...)
  → 모든 변수를 동시에 제어 불가
  → "최악의 경우" 계산 불가 = WCRT 무한대
  → IEC 61508 인증 요구사항 충족 불가

TSN으로 WCRT 계산 가능:
  TAS (802.1Qbv) 사전 스케줄링:
  → 각 프레임의 전송 시간 사전에 결정
  → 수학적으로 WCRT 계산 가능
  → 인증 근거 문서 제공 가능
```

### 5.2 패킷 손실의 안전 영향

```
제어 패킷 손실 시나리오 (1% 손실률 가정):

사이클 타임: 1ms, 손실률: 1%
→ 평균 100 사이클마다 1회 손실
→ 100ms마다 1ms 제어 공백

실제 손실은 균등하지 않음 (Burst Loss):
  혼잡 발생 시 연속 10~20 패킷 손실 가능
  → 10~20ms 제어 공백

10ms 제어 공백의 영향:
  로봇 팔 끝단 속도: 0.5 m/s
  10ms × 0.5 m/s = 5mm 이동 (제어 없이)
  → 수술 중 5mm 무제어 이동 → 조직 관통/손상 가능

의료기기 규제 요구:
  FDA 21 CFR Part 820: 위험 분석 및 완화 요구
  IEC 62304 Class C: 제어 소프트웨어 무결성 수준 최고
  → 패킷 손실이 안전에 영향을 미치면 손실 = 0 이어야 함
  → Best-Effort Ethernet으로는 이 요구 충족 불가
```

## 6. TSN으로의 전환: 문제 해결의 로드맵

### 6.1 TSN이 해결하는 문제

```
비결정성 원인별 TSN 해결책:

[원인 1: 큐잉 지연]
  TSN TAS (IEEE 802.1Qbv - Time-Aware Shaper):
  → 사전에 계산된 게이트 스케줄로 타임슬롯 할당
  → 스케줄된 트래픽(ST)은 정해진 시간에만 전송
  → 큐잉 지연 = 0 (사전 계획으로 혼잡 방지)

[원인 2: HOL Blocking]
  TSN Frame Preemption (IEEE 802.1Qbu + 802.3br):
  → 낮은 우선순위 대형 프레임을 중간에 선점 가능
  → 긴급 소형 프레임: 대기 없이 즉시 전송
  → HOL Blocking 최대 지연 = mPacket 최소 크기 전송 시간(수십 ns)

[원인 3: Broadcast 폭풍]
  TSN PSFP (IEEE 802.1Qci - Per-Stream Filtering and Policing):
  → 각 트래픽 흐름에 대해 대역폭 제한
  → 폭발적 브로드캐스트: 임계값 초과 즉시 차단
  → Babbling Idiot 방어

[원인 4: 클록 비동기]
  TSN gPTP (IEEE 802.1AS - Generalized PTP):
  → 네트워크 전체 µs 수준 시간 동기화
  → TAS 스케줄링의 시간 기준 통일
  → 노드 간 타임스탬프 비교 가능

[원인 5: 패킷 손실]
  TSN FRER (IEEE 802.1CB - Frame Replication and Elimination):
  → 동일 패킷을 두 경로로 복제 전송
  → 한 경로 장애 시 다른 경로 자동 사용
  → 이론적 손실률 = 두 경로 동시 장애 확률 (매우 낮음)
```

### 6.2 TSN 이전과 이후의 지연 분포 비교

```
Before TSN (Best-Effort, 1Gbps, 50% 배경 부하):
  지연 분포: 12 µs ~ 8,000 µs (660배 변동)
  제어 불가능: 사전에 최악값 계산 불가

After TSN (TAS 적용, 동일 환경):
  제어 트래픽 지연 분포: 14 µs ~ 18 µs (±4 µs 범위!)
  최악 지연 사전 계산: 수학적 보장 가능
  배경 트래픽: 14 µs ~ 8,000 µs (여전히 변동, 하지만 제어 트래픽 격리됨)

→ 동일 물리 Ethernet 인프라에서 TSN으로 결정론 달성
→ 추가 하드웨어 없이 소프트웨어/설정 변경으로 구현 가능
```

### 6.3 TSN 없이 결정론을 얻는 다른 방법들

```
대안 1: 전용 물리 네트워크 분리 (구식 접근)
  안전 제어 전용 네트워크 + 일반 데이터 네트워크를 완전 분리
  장점: 물리적 격리 → 간섭 없음
  단점: 비용 2배, 유연성 없음, 소프트웨어 중심 설계 불가

대안 2: EtherCAT (Ethernet 활용 전용 Fieldbus)
  장점: 100µs 이하 사이클 타임 달성
  단점: 독자 프로토콜 → 표준 IP 스택 미사용 → 벤더 종속

대안 3: 대역폭 과잉 프로비저닝
  링크 속도를 10Gbps로 올려 혼잡 자체를 방지
  장점: 비교적 구현 쉬움
  단점: 비용 ↑, 비결정성 완전 제거 불가, Jitter는 여전히 존재

→ TSN이 유일하게 표준 Ethernet 생태계를 유지하면서
  하드 실시간 요구사항을 충족하는 방법
```

## 7. 3장 TSN으로의 연결

```
이 문서에서 확인한 핵심 문제들과 3장에서의 해결책:

문제                    → 3장의 해결책
────────────────────────────────────────────────────────────────
큐잉 지연 (비결정성)    → IEEE 802.1AS (시간 동기화)
                        → IEEE 802.1Qbv (TAS: 타임슬롯 스케줄링)
HOL Blocking           → IEEE 802.1Qbu (Frame Preemption)
패킷 손실              → IEEE 802.1CB (FRER: 경로 이중화)
Babbling Idiot         → IEEE 802.1Qci (PSFP: 스트림 필터링)
클록 비동기            → IEEE 802.1AS (gPTP: 정밀 시간 동기화)
CBS (비율 보장)        → IEEE 802.1Qav (Credit-Based Shaping)

1장에서 배운 것들과 TSN의 연결:
  OSI L2            → TSN은 모두 L2에서 동작
  Ethernet 프레임   → TAS는 프레임 단위로 게이트 제어
  VLAN PCP          → TSN은 PCP 값으로 트래픽 클래스 분류
  QoS 큐            → TSN은 큐별 게이트 스케줄 설정
  직렬화 지연       → Guard Band 계산의 핵심 파라미터

3장 학습 목표:
  ✓ gPTP로 네트워크 전체 시간 동기화하는 방법
  ✓ TAS GCL(Gate Control List) 설계 및 설정
  ✓ CBS 파라미터 계산 및 설정
  ✓ Frame Preemption 설정 및 효과 확인
  ✓ FRER 이중 경로 설정
  ✓ Linux에서 tc taprio, tc cbs 명령어 실습
```
