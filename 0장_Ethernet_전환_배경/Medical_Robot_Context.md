# 0.4 의료 로봇 맥락: 왜 네트워크가 생사를 가르는가

## 개요

의료 로봇은 단순한 자동화 장비가 아닙니다.
수술실에서 로봇의 네트워크 지연 1ms가 외과의의 손 움직임을 그대로 집도 부위에 전달하지 못하게 만들고,
10ms 이상의 지연은 의사가 실제로 **촉각적 부자연스러움(haptic artefact)** 을 느끼게 합니다.
그리고 100ms를 초과하면 인간은 그것을 **"조종 불능"** 으로 인식합니다.

이 절은 다음 질문에 답합니다:

> **"왜 의료 로봇에서 Ethernet 네트워크의 결정성(determinism)이 수술 품질과 환자 안전에 직결되는가?"**

---

## 1. 의료 로봇의 종류와 네트워크 요구사항

### 1.1 분류별 특성

```
┌─────────────────────────────────────────────────────────────────────┐
│                   의료 로봇 분류와 네트워크 요구                      │
├──────────────┬──────────────┬────────────┬────────────┬─────────────┤
│  분류        │  대표 제품   │ 제어 주기  │ 허용 지연  │ 허용 지터   │
├──────────────┼──────────────┼────────────┼────────────┼─────────────┤
│ 복강경 수술  │ da Vinci Xi  │ 1,000 Hz   │ < 1 ms     │ < ±50 µs   │
│ 정형외과     │ MAKO, ROSA   │ 500 Hz     │ < 2 ms     │ < ±100 µs  │
│ 방사선 치료  │ CyberKnife   │ 100 Hz     │ < 10 ms    │ < ±500 µs  │
│ 재활 로봇    │ Lokomat      │ 100 Hz     │ < 10 ms    │ < ±1 ms    │
│ 내시경 로봇  │ Flex, Monarch│ 200 Hz     │ < 5 ms     │ < ±200 µs  │
│ 원격 수술    │ 연구용 시스템│ 1,000 Hz   │ < 0.5 ms   │ < ±10 µs   │
└──────────────┴──────────────┴────────────┴────────────┴─────────────┘
```

> **참고**: da Vinci 시스템은 1kHz 제어 루프에서 조종사 손의 위치를 1mm 미만 정확도로 추적합니다.
> 네트워크 지연이 제어 주기(1ms)를 초과하면 제어 알고리즘의 안정성 보장이 불가능합니다.

### 1.2 복강경 수술 로봇의 실시간 데이터 흐름

```
외과의 콘솔                            환자 카트

│ (6-DOF 위치/힘)  │  1,000 패킷/s   │ 각 관절: 위치 + 토크 + 힘    │
│                  │ ←───────────── │ 엔드이펙터: 그리퍼 제어      │
│ 햅틱 피드백      │   힘 피드백     └──────────────────────────────┘
│ (force/torque)  │                          │
│                  │    ┌────────────────────┘
│ 3D 내시경 화면   │ ←──│ 4K 듀얼 카메라 스트림 (~3 Gbps)
└──────────────────┘    │ 초음파 센서 데이터
                        │ 관절 인코더 (12개 × 32-bit)
```

|---|---|---|---|

| 관절 토크 피드백 (12축 × 8byte) | 96 byte | 1 ms | 768 Kbps |

| 안전 상태 heartbeat | 64 byte | 1 ms | 512 Kbps |
| **합계 (제어 데이터만)** | | | **~3.8 Mbps** |

## 2. Latency가 수술 품질에 미치는 영향

### 2.1 인간 지각 임계값과 수술 요구사항

인간 신경계의 반응 시간 연구에 따르면:

```
촉각 지각 임계값:
  1 ms  이하: 완전 투명 (인지 불가)
  1~10 ms  : 훈련된 외과의가 감지 가능 (haptic artefact 시작)
  10~50 ms : 대부분 외과의가 감지 (제어감 저하)
  50~100 ms: 명확한 조종 불안정감
  100 ms+  : 비직관적, 안전하지 않은 조종

시각 피드백 임계값:
  < 16 ms: 60fps 수준, 자연스러운 화면
  16~33 ms: 약간의 지연감
  33~100 ms: 명확한 지연, 정밀 작업 저하
  100 ms+: 작업 불가 수준
```

### 2.2 지연-제어 안정성 관계 (제어공학적 분석)

**PID 제어에서 통신 지연의 영향:**

```python
# 이산 PID에 통신 지연이 추가될 때 위상 여유 감소
# 제어 주기: T = 1ms (1kHz)
# 통신 지연: Td (가변)

# 위상 지연 = Td / T × 360°
T = 1e-3  # 1ms

delays = {
    "0.1ms": 0.1e-3,   # TSN 달성 가능
    "1ms":   1e-3,     # Best-Effort 최소
    "5ms":   5e-3,     # Best-Effort 평균
    "10ms":  10e-3,    # Best-Effort 최악
}

for name, Td in delays.items():
    phase_delay = (Td / T) * 360
    print(f"지연 {name}: 위상 지연 = {phase_delay:.0f}°")

# 출력:
# 지연 0.1ms: 위상 지연 = 36°     ← 안정 마진 충분
# 지연 1ms:   위상 지연 = 360°    ← 불안정 경계
# 지연 5ms:   위상 지연 = 1800°   ← 완전 불안정
# 지연 10ms:  위상 지연 = 3600°   ← 발진(oscillation) 확실
```

> **핵심**: 1kHz 제어 루프에서 1ms 이상의 통신 지연은 이미 위상 여유를 소진합니다.
> 제어 이득을 낮추면 안정성은 회복되지만, 추종 성능(tracking performance)이 떨어집니다.

### 2.3 지연에 따른 물리적 오차

복강경 수술 로봇에서 10cm/s 속도로 움직이는 경우:

```
물리적 위치 오차 = 속도 × 지연

속도: 10 cm/s = 100 mm/s

지연     → 위치 오차    → 임상적 영향
0.1 ms  → 0.01 mm     ← 무시 가능 (봉합사 두께 정도)
1 ms    → 0.1 mm      ← 미세 수술 한계 (허용 가능)
5 ms    → 0.5 mm      ← 혈관 봉합 정밀도 초과
10 ms   → 1.0 mm      ← 중요 구조물 손상 위험 영역
50 ms   → 5.0 mm      ← 명확한 수술 오류
100 ms  → 10 mm       ← 심각한 조직 손상 가능
```

> **수술 기준**: 복강경 수술에서 봉합 정밀도 요구사항은 일반적으로 ±0.5mm 이내입니다.
> 이는 5ms 미만의 제어 지연을 요구하며, TSN 없이는 달성이 어렵습니다.

### 2.4 Jitter가 Force/Torque Control에 미치는 영향

임피던스 제어(impedance control)는 수술 로봇의 핵심 제어 방식입니다.
조직에 가해지는 힘을 정밀하게 제어하여 불필요한 절개나 손상을 방지합니다.

```
임피던스 제어 방정식:
  F_interaction = M_d × ẍ_e + B_d × ẋ_e + K_d × x_e

  M_d: 희망 질량 (virtual mass)
  B_d: 희망 감쇠 (virtual damping)
  K_d: 희망 강성 (virtual stiffness)
  x_e: 위치 오차

B_d (감쇠항)은 속도 차분으로 계산:
  ẋ_e ≈ (x_e[k] - x_e[k-1]) / T_sample

T_sample에 jitter가 있으면:
  T_actual = T_nominal ± Δt (jitter)
  ẋ_e_estimated ≠ ẋ_e_actual

B_d = 1000 N·s/m, Jitter = ±1ms일 때 힘 오차:
  F_error = B_d × |x_e| × |ΔT| / T²

  예: |x_e| = 1mm, ΔT = 1ms, T = 1ms
  F_error = 1000 × 0.001 × 0.001 / 0.000001 = 1.0 N
```

> **1.0 N의 힘 오차는 조직 손상 임계값(약 2-5N)의 20-50%에 해당합니다.**
> Jitter ±1ms → 힘 제어 오차 최대 1N → 반복적 수술에서 조직 손상 누적.

### 2.5 패킷 손실의 영향

```
패킷 손실 시나리오 (1kHz 제어 루프):
┌─────────────────────────────────────────────────────┐
│ 정상 패킷 흐름:                                      │
│ ●─●─●─●─●─●─●─●─●─●─●─●─●─●─●  (1ms 간격)       │
│                                                     │
│ 1% 손실 (랜덤):                                      │
│ ●─●─●─○─●─●─●─●─●─○─●─●─●─●─●  (○ = 손실)       │
│          ↑                                          │
│          2ms gap → Hold-Last-Command                │
│          ↑                                          │
│ 버스트 손실 (Best-Effort 스위치):                    │
│ ●─●─●─○─○─○─○─○─●─●─●─●─●─●─●  (5-packet burst) │
│          └───────────┘                              │
│          5ms gap → 0.5mm uncontrolled movement      │
└─────────────────────────────────────────────────────┘
```

**IEC 62304 관점에서 패킷 손실 처리 전략:**

| 전략 | 설명 | 위험도 | 적용 상황 |
|---|---|---|---|
| Hold-Last-Command | 마지막 명령 유지 | 낮음 | 짧은 갭 (< 2ms) |
| Fail-Safe Position | 안전 위치로 이동 | 중간 | 중간 갭 (2~10ms) |
| Emergency Stop | 즉시 정지 | 높음 | 긴 갭 (> 10ms) |
| 이중화 경로 (FRER) | 대체 경로 자동 전환 | 매우 낮음 | 미션 크리티컬 |

## 3. Safety-Critical System으로서의 의료 로봇

### 3.1 Safety-Critical System 정의

```
Safety-Critical System: 오작동 시 인명 피해, 심각한 부상,
또는 환경적 재앙을 초래할 수 있는 시스템

의료 로봇의 safety-critical 속성:
┌─────────────────────────────────────────────────────┐
│  Level 1: 즉각적 위험 (네트워크 직접 관련)            │
│  ● 로봇 팔 오작동 → 조직 절개, 혈관 손상             │
│  ● 햅틱 피드백 손실 → 과도한 힘 적용               │
│  ● 제어 신호 지연 → 로봇 의도치 않은 움직임          │
│                                                     │
│  Level 2: 간접 위험 (시스템 레벨)                    │
│  ● 전원 시스템 통신 오류 → 전기 수술 기구 오작동     │
│  ● 마취 기기 네트워크 분리 → 마취 감시 중단          │
│  ● EMR 시스템 지연 → 환자 정보 불일치              │
│                                                     │
│  Level 3: 운영 위험 (수술실 환경)                    │
│  ● 방해 전파 → 통신 단절                           │
│  ● 사이버 공격 → 원격 조종 가능성                  │
│  ● 전원 글리치 → 순간 네트워크 재시작               │
└─────────────────────────────────────────────────────┘
```

### 3.2 Fail-Safe vs Fail-Secure vs Fail-Operational

```
의료 로봇 네트워크 장애 시 요구 동작:

┌─────────────────┬────────────────────────────────────────────────┐
│ Fail-Safe       │ 안전한 상태로 전환 (로봇 정지, 최소 침습 자세) │
│                 │ → 대부분의 네트워크 장애에 적용                │
│                 │ 예: 링크 다운 50ms → 비상 정지                │
├─────────────────┼────────────────────────────────────────────────┤
│ Fail-Secure     │ 보안 상태 유지 (원격 접근 차단, 로컬만 허용)   │
│                 │ → 사이버 공격, 외부 네트워크 장애에 적용       │
│                 │예: 방화벽 다운 → 모든 외부 트래픽 차단 (deny) │
├─────────────────┼────────────────────────────────────────────────┤
│ Fail-Operational│ 장애 상황에서도 운영 계속                      │
│                 │ → 원격 수술 시 이중화 경로에서 적용            │
│                 │ 예: 1차 링크 다운 → 50µs 내 2차 경로 전환    │
│                 │ (FRER IEEE 802.1CB 사용)                      │
└─────────────────┴────────────────────────────────────────────────┘

수술 단계별 요구사항:
  수술 전: Fail-Safe (언제든 정지 가능)
  수술 중 비위험 단계: Fail-Safe
  수술 중 위험 단계 (절개, 봉합): Fail-Operational
  수술 종료: Fail-Safe
```

## 4. IEC 62304: 의료기기 소프트웨어 생명주기 표준

### 4.1 표준 개요

**IEC 62304:2006 + AMD1:2015** (의료기기 소프트웨어 생명주기 프로세스)

```
목적: 의료기기에 포함된 소프트웨어의 개발, 유지보수, 위험 관리를
      위한 요구사항 정의

적용 대상:
  ● 의료기기에 내장된 소프트웨어 (embedded SW)
  ● 의료기기 그 자체인 소프트웨어 (SW as medical device, SaMD)
  ● 의료기기를 구동하는 소프트웨어 인프라 (네트워크 미들웨어 포함)

관련 표준:
  ● IEC 62304 ← 소프트웨어 생명주기
  ● ISO 14971  ← 위험 관리
  ● IEC 60601  ← 의료 전기 장비 안전
  ● IEC 62443  ← 사이버 보안
  ● ISO 13485  ← 품질 관리 시스템
```

### 4.2 소프트웨어 안전 등급 (Software Safety Class)

```
┌─────────┬──────────────────────────────────┬─────────────────────┐
│  등급   │  정의                             │  네트워크 영향       │
├─────────┼──────────────────────────────────┼─────────────────────┤
│ Class A │ 장애가 상해를 유발하지 않음       │ 일반 모니터링 데이터│
│         │ (부상 없음)                       │ 환자 기록, 로그     │
├─────────┼──────────────────────────────────┼─────────────────────┤
│ Class B │ 장애가 경상을 유발할 수 있음      │ 진단 이미지, 알람   │
│         │ (비심각한 부상)                   │ 약물 투여 계산      │
├─────────┼──────────────────────────────────┼─────────────────────┤
│ Class C │ 장애가 중상 또는 사망을 유발할   │ 로봇 제어 신호      │
│         │ 수 있음 (심각한 부상 또는 사망)  │ 안전 인터락 상태    │
│         │                                  │ 전기 수술 제어      │
└─────────┴──────────────────────────────────┴─────────────────────┘
```

### 4.3 Class C 소프트웨어의 네트워크 요구사항

IEC 62304 Class C는 가장 엄격한 요구사항을 가집니다.
의료 로봇 제어 네트워크는 Class C에 해당합니다.

**주요 요구사항:**

```
1. 완전한 소프트웨어 요구사항 명세 (§5.2)
   → 네트워크 타이밍 요구사항 명문화 필수
   → "1ms 이내 전달" 같은 정량적 명세 필요

2. 소프트웨어 아키텍처 설계 (§5.3)
   → 네트워크 계층 아키텍처 문서화
   → 안전-관련 / 비안전 데이터 분리 설계

3. 위험 관리 통합 (ISO 14971 연계, §4.2)
   → 네트워크 장애 FMEA (Failure Mode and Effect Analysis)
   → 각 네트워크 장애 시나리오의 위험도 계산:
     위험도 = 발생 가능성 × 심각도

4. 소프트웨어 검증 및 유효성 확인 (§5.7, §5.8)
   → 네트워크 지연 측정 테스트 (실환경)
   → Worst-Case 패킷 손실 주입 테스트
   → 결정론적 거동 증명 (WCRT analysis)

5. 소프트웨어 유닛 테스트 (§5.5)
   → 통신 스택 단위 테스트
   → 경계 조건 (최대 지연, 최대 손실) 테스트

6. 형상 관리 (§8)
   → 네트워크 설정 파일 버전 관리
   → 스위치 펌웨어, GCL (Gate Control List) 버전 추적

7. 문제 해결 프로세스 (§9)
   → 네트워크 관련 필드 이슈 추적 프로세스
   → 패치 배포 시 재검증 절차
```

### 4.4 ISO 14971 위험 관리와 네트워크 FMEA

```
네트워크 FMEA 예시 (수술 로봇 제어망):

장애 모드           발생  심각도  위험도  완화 조치
─────────────────────────────────────────────────────
링크 단선           2     5       10      FRER 이중화
패킷 지연 > 5ms     3     4       12      TAS + 지연 모니터링
PTP 동기화 손실     2     5       10      GMC 이중화
VLAN 설정 오류      1     5       5       설정 검증 자동화
패킷 손실 > 0.1%   2     4       8       QoS + 손실 모니터링
스위치 재시작       1     5       5       WDT + 빠른 복구 설계
────────────────────────────────────────────────────
발생: 1(거의없음) ~ 5(자주), 심각도: 1(경미) ~ 5(치명)
위험도 > 10: 추가 완화 조치 필수 (IEC 62304 Class C 요건)
```

```
  ● MDR 2017/745: Medical Device Regulation

  ● EN 62304: IEC 62304의 유럽 채택 버전
  ● ENISA: 의료기기 사이버보안 가이드라인

  ● AI/ML 기반 의료기기 허가심사 가이드라인
```

### 5.2 네트워크 관련 규제 요구사항

**FDA 2023 사이버보안 가이드라인에서 네트워크 관련 핵심 요구사항:**

```
1. SBOM (Software Bill of Materials):
   → 네트워크 스택, 드라이버, 미들웨어 모든 구성요소 목록화
   → 알려진 취약점 (CVE) 추적 의무

2. 인증 및 접근 제어:
   → 의료기기 네트워크에서 기기 인증 (802.1X)
   → 최소 권한 원칙 적용

3. 데이터 보호:
   → 전송 중 암호화 (TLS 1.3+)
   → 환자 데이터 PHI 보호 (HIPAA 연계)

4. 업데이트 메커니즘:
   → 서명된 OTA 업데이트
   → 롤백 기능 구비

5. 모니터링:
   → 이상 트래픽 감지 (IDS/IPS)
   → 감사 로그 (audit log) 유지
```

### 5.3 IEC 62443: 산업 사이버보안과 의료 융합

```
IEC 62443 (산업 자동화 제어 시스템 사이버보안):
의료 로봇은 산업 로봇과 의료기기의 교차점에 있어 양쪽 적용

보안 레벨 (Security Level):
  SL 1: 우발적 위반 방어
  SL 2: 의도적 단순 공격 방어  ← 대부분 의료 로봇 목표
  SL 3: 정교한 공격자 방어
  SL 4: 국가급 공격자 방어

네트워크 분리 (Zone & Conduit 모델):
┌─────────────────────────────────────────────────────┐
│  Zone 1: 병원 IT 네트워크 (일반 LAN)                │
│    └──[방화벽/DMZ]──                                │
│  Zone 2: 의료기기 네트워크 (분리된 VLAN)             │
│    └──[게이트웨이]──                                │
│  Zone 3: 수술 로봇 제어 네트워크 (TSN, 고립)         │
│    └── 직접 외부 접속 없음                           │
└─────────────────────────────────────────────────────┘
```

## 6. 실제 사례: 네트워크 장애가 수술에 미치는 영향

### 6.1 알려진 의료 로봇 통신 사고 패턴

(아래 내용은 공개된 FDA MAUDE 데이터베이스 및 학술 연구 기반)

```
사례 유형 1: 일시적 통신 중단
  발생: 전기 수술 기구에서 EMI 발생 → 10ms 통신 중단
  영향: 로봇 팔 제어 신호 손실 → 자동 안전 정지
  결과: 수술 일시 중단, 재시작 필요
  교훈: EMI 차폐 설계, 실드 케이블, 분리 전원 필요

사례 유형 2: 카메라/제어 네트워크 혼재
  발생: 4K 영상 트래픽이 제어 패킷 지연 유발
  영향: 제어 지연 1ms → 7ms 증가
  결과: 외과의 햅틱 피드백 저하, 정밀 작업 어려움
  교훈: QoS 분리, VLAN 기반 트래픽 격리 필수

사례 유형 3: 소프트웨어 업데이트 후 타이밍 이슈
  발생: OS 업데이트 후 네트워크 드라이버 변경 → 지터 증가
  영향: ±50µs → ±2ms로 지터 증가
  결과: Class C 요건 위반, 임시 사용 중지
  교훈: 네트워크 성능 회귀 테스트 필수
```

### 6.2 원격 수술에서의 네트워크 요구사항

원격 수술(telesurgery)은 가장 극단적인 네트워크 요구사항을 가집니다.

```
5G 네트워크를 이용한 원격 수술 시험 결과 (연구 논문 기반):

거리 10km (동일 도시):
  RTT 지연: 1~3ms → 가능 (연구 수준)
  지터: ±0.5ms → 경계 수준
  패킷 손실: < 0.01% → 허용 가능

거리 100km (도시간):
  RTT 지연: 10~30ms → 위험 (경험 외과의 한계 초과)
  지터: ±2ms → 부적합
  패킷 손실: < 0.1% → 버스트 손실 허용 불가

거리 1000km+ (국제):
  물리적 빛의 속도 한계: ~5ms (광섬유 기준)
  RTT 최소값: ~10ms (왕복)
  → 현재 기술로 안전한 원격 수술 불가
  → 실시간 햅틱 피드백 포기 또는 반자율 모드 필요

결론: 원격 수술의 지리적 한계는 네트워크 기술보다
      빛의 속도라는 물리 법칙에 의해 결정됨
```

## 7. 의료 로봇 네트워크 아키텍처 설계 원칙

### 7.1 계층적 네트워크 분리

```
수술실 네트워크 아키텍처 (권장 구조):

┌─────────────────────────────────────────────────────────────────┐
│                     병원 백본 네트워크                            │
│                    (일반 IT, 10GbE)                             │
└──────────────────────────┬──────────────────────────────────────┘
                           │ 방화벽 / DMZ
┌──────────────────────────▼──────────────────────────────────────┐
│                   의료기기 네트워크 VLAN                          │
│               (의료 장비 연동, 1GbE, QoS 적용)                  │
│    모니터링 시스템 │ EMR 인터페이스 │ 영상 장비 │ 마취 기기      │
└──────────────────────────┬──────────────────────────────────────┘
                           │ TSN 게이트웨이 스위치
┌──────────────────────────▼──────────────────────────────────────┐
│              수술 로봇 TSN 제어 네트워크                          │
│          (IEEE 802.1Qbv TAS, 1GbE/2.5GbE, 격리)               │
│  로봇 팔 A │ 로봇 팔 B │ 로봇 팔 C │ 햅틱 콘솔 │ 안전 인터락  │
│                                                                 │
│  최대 지연: 500µs    지터: ±10µs    손실: < 10^-9               │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 네트워크 안전 요구사항 체크리스트

```
□ 네트워크 계층 분리 (TSN 제어망 ↔ 일반 IT망)
□ 제어 트래픽 QoS 최우선 (PCP 7)
□ EEE 비활성화 (모든 TSN 링크)
□ PTP 하드웨어 타임스탬프 (정밀도 < 100ns)
□ FRER 이중화 (경로 장애 50µs 내 복구)
□ Watchdog Timer (네트워크 침묵 감지, 50ms 임계값)
□ VLAN 기반 트래픽 격리 (안전/비안전 분리)
□ EMI 차폐 케이블 (STP Cat6A 이상)
□ 감사 로그 (네트워크 이벤트 전체 기록)
□ 정기적 네트워크 성능 검증 (분기별 WCRT 측정)
```

## 8. 요약: 의료 로봇을 위한 네트워크 철학

```
일반 IT 네트워크의 철학:
  "평균적으로 빠르게, 가끔 느려도 재전송으로 해결"
  → 사용자 경험에서 허용 가능

의료 로봇 제어 네트워크의 철학:
  "항상 정해진 시간 내에, 단 한 번의 예외도 없이"
  → 결정론적 보장 없이는 안전 보장 불가능

┌─────────────────────────────────────────────────────┐
│ 핵심 요구사항 계층:                                  │
│                                                     │
│  1. Determinism (결정성)                            │
│     → 모든 패킷이 예측 가능한 시간 내 도착           │
│     → TSN TAS (802.1Qbv)로 달성                    │
│                                                     │
│  2. Reliability (신뢰성)                            │
│     → 패킷 손실 없음 (또는 < 10^-9)                │
│     → FRER (802.1CB)로 이중화                      │
│                                                     │
│  3. Security (보안)                                 │
│     → 외부 접근 차단, 무결성 보장                   │
│     → VLAN 분리 + TLS + 802.1X 인증                │
│                                                     │
│  4. Regulatory Compliance (규제 준수)               │
│     → IEC 62304, ISO 14971, FDA 가이드라인           │
│     → 문서화, 검증, 추적성 확보                     │
└─────────────────────────────────────────────────────┘
```
