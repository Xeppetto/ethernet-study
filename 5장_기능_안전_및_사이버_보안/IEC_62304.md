# IEC 62304 (의료기기 소프트웨어 수명 주기)

## 개요
IEC 62304:2006+AMD1:2015는 의료기기 소프트웨어의 개발, 유지보수, 위험 관리에 관한 국제 표준입니다. FDA(미국 식품의약국)와 EU MDR(Medical Device Regulation) 모두 의료기기 소프트웨어에 IEC 62304 준수를 요구합니다. 수술 로봇, 진단 장비, 환자 모니터링 시스템 등 의료기기 내 소프트웨어에 적용됩니다.

**ISO 26262와의 관계**:
- ISO 26262: 자동차 E/E 시스템 (ASIL A~D)
- IEC 62304: 의료기기 소프트웨어 (Class A/B/C)
- 공통 원칙: V-Model, 위험 기반 설계, 추적성

## 소프트웨어 안전 등급 (Software Safety Classification)

```
IEC 62304 Class 분류:

Class A: 부상 없음
  - 소프트웨어 결함이 부상을 일으킬 수 없음
  - 예: 병원 관리 소프트웨어, 보고서 생성
  - 요구 수준: 최소 (기본 개발 프로세스)

Class B: 심각하지 않은 부상
  - 소프트웨어 결함으로 경미한 부상 가능
  - 예: 환자 모니터링 (알람 기능), 재활 기기 제어
  - 요구 수준: 중간 (상세 설계, 단위 테스트)

Class C: 사망 또는 심각한 부상
  - 소프트웨어 결함으로 사망 또는 중상 가능
  - 예: 수술 로봇 제어, 방사선 치료 제어, 마취 기기
  - 요구 수준: 최고 (전체 IEC 62304 요구사항 적용)

Class 결정 프로세스 (IEC 14971 연계):
  ┌──────────────────────────────────────────────────┐
  │  IEC 14971 RISK MANAGEMENT                       │
  │  ① 위험원 식별 (Hazard Identification)            │
  │  ② 위험 추정 (Risk Estimation): 심각도 × 발생확률 │
  │  ③ 위험 평가 (Risk Evaluation)                   │
  │  ④ 위험 통제 (Risk Control) → SW 안전 기능        │
  └─────────────────────────────────────────────────┘
           │ 위험 통제에 SW가 사용될 때
           ▼
  SW 안전 등급 (Class A/B/C) 결정
```

## 요구사항 추적성 매트릭스 (RTM)

```
RTM (Requirements Traceability Matrix) 예시:

┌────────┬─────────────────────────┬──────────┬────────────┬────────────┐
│ 요구사항│ 설명                    │ 설계     │ 구현       │ 테스트     │
│ ID     │                         │ 문서     │ 파일       │ 케이스 ID  │
├────────┼─────────────────────────┼──────────┼────────────┼────────────┤
│ SRS-01 │ 관절 속도 < 500mm/s     │ ARCH-001 │ pid.c:145  │ TC-JC-001  │
│ SRS-02 │ 힘 제한 < 10N           │ ARCH-002 │ force.c:89 │ TC-FM-001  │
│ SRS-03 │ E-Stop 응답 < 10ms      │ ARCH-003 │ estop.c:22 │ TC-ES-001  │
│ SRS-04 │ 통신 손실 > 100ms → 정지│ ARCH-004 │ comm.c:210 │ TC-CM-001  │
│ SRS-05 │ DTC 기록 비휘발성 저장  │ ARCH-005 │ dtc.c:67   │ TC-DG-001  │
└────────┴─────────────────────────┴──────────┴────────────┴────────────┘

추적성 방향:
  요구사항 → 설계 → 코드 → 테스트 (순방향 추적)
  테스트 → 코드 → 설계 → 요구사항 (역방향 추적)
  양방향 완전성 확인 → 심사 합격 요건
```

## 코드 커버리지 요구사항

```
IEC 62304 Class별 커버리지:

Class A: 요구 없음 (자율)

Class B:
  - Statement Coverage: 모든 코드 라인 실행
  - 목표: 100% (실무: ≥ 90%)

Class C: 더 엄격
  - Statement Coverage: 100%
  - Branch Coverage: 모든 분기 실행 (true/false)
  - MC/DC (Modified Condition/Decision Coverage):
    각 조건(Condition)이 결정(Decision)에 독립적으로
    영향을 미침을 확인

MC/DC 예시:
  코드: if (speed > limit && emergency_stop == false)

  조건 A: speed > limit
  조건 B: emergency_stop == false
  결정 D: A && B

  MC/DC 테스트 케이스:
  ┌───┬───┬───┬──────────────────────────────────────┐
  │ A │ B │ D │ 설명                                 │
  ├───┼───┼───┼──────────────────────────────────────┤
  │ T │ T │ T │ 기본 케이스                           │
  │ F │ T │ F │ A 변화 → D 변화 (A의 독립 영향 증명)  │
  │ T │ F │ F │ B 변화 → D 변화 (B의 독립 영향 증명)  │
  └───┴───┴───┴──────────────────────────────────────┘
  → 3개 테스트로 MC/DC 달성

커버리지 도구:
  gcov + lcov: C/C++ 커버리지
  pytest-cov:  Python 커버리지
  LDRA Testbed: 의료기기 인증 도구
  VectorCAST:  자동차/의료 임베디드 전용
```

## SOUP (Software of Unknown Pedigree)

```
SOUP 관리 (IEC 62304 Section 8):

SOUP 정의:
  - 의료기기 개발을 목적으로 개발되지 않은 SW 컴포넌트
  - 오픈소스 라이브러리, COTS(Commercial Off-The-Shelf)
  - 예: Linux Kernel, ROS 2, OpenSSL, FastDDS

SOUP 관리 요구사항:
  ① 식별: 이름, 버전, 제조사, 라이선스
  ② 기능 요구사항: SOUP이 제공해야 할 기능 정의
  ③ 비기능 요구사항: 성능, 보안 등
  ④ 검증: SOUP이 의도대로 동작함을 확인하는 테스트
  ⑤ 알려진 결함: CVE, 버그 목록 확인 및 영향 평가
  ⑥ 위험 관리: SOUP 결함이 안전에 미치는 영향 분석

수술 로봇 SOUP 목록:
  ┌──────────────────┬─────────────┬──────────┬──────────────┐
  │ SOUP 이름        │ 버전        │ 용도     │ Safety Class │
  ├──────────────────┼─────────────┼──────────┼──────────────┤
  │ Linux Kernel     │ 5.15.83-rt  │ RTOS     │ Class B      │
  │ ROS 2 (Humble)  │ 0.10.0      │ 미들웨어  │ Class B      │
  │ FastDDS          │ 2.9.1       │ DDS 통신  │ Class B      │
  │ OpenSSL          │ 3.0.8       │ TLS 보안  │ Class B      │
  │ libEigen         │ 3.4.0       │ 행렬 연산  │ Class C      │
  │ Qt 6             │ 6.4.2       │ UI       │ Class A      │
  └──────────────────┴─────────────┴──────────┴──────────────┘
```

## 정적 분석 및 코딩 표준

```bash
# MISRA C:2012 체크 (PC-lint Plus 또는 Cppcheck)
cppcheck --enable=all \
         --addon=misra.py \
         --suppressions-list=misra_suppressions.txt \
         src/

# Clang Static Analyzer
scan-build --use-analyzer clang \
           make -C build/

# 의료기기 전용 도구
# LDRA Testbed: 커버리지 + MISRA + 복잡도
# Parasoft C/C++test: 자동화 검증
# CodeSonar: 심층 취약점 분석

# 코드 복잡도 분석 (McCabe Cyclomatic Complexity)
# IEC 62304 권장: 복잡도 ≤ 10 (Class C는 더 엄격)
lizard src/ --CCN 10 --length 50 --arguments 10

# 출력 예시:
# ================================================
# function: pid_controller @ pid.c:145
#   Cyclomatic Complexity: 8 (OK)
# function: trajectory_planner @ traj.c:89
#   Cyclomatic Complexity: 15 (WARNING - 초과)
# ================================================
```

## 변경 관리 (Change Management)

```
IEC 62304 Section 6.2 - SW 변경 관리:

변경 영향 분석 절차:
  ① 변경 요청 (Change Request) 등록
  ② 영향 범위 분석 (Impact Analysis)
     - 변경된 SW Item/Unit 식별
     - 관련 요구사항 확인 (RTM)
     - 안전 등급 재평가 필요 여부 판단
  ③ 위험 재평가 (IEC 14971 연계)
     - 새로운 위험 또는 기존 위험 변화?
  ④ 재테스트 범위 결정
     - 전체 회귀 테스트 vs 영향 범위만
  ⑤ 검토 및 승인 (Change Control Board)
  ⑥ 구현 및 검증
  ⑦ 출시 및 기록

변경 관리 도구:
  Jira + Jira-GitHub 연동: 이슈 추적
  Polarion ALM: 의료기기 특화 요구사항/변경 관리
  IBM DOORS: 대규모 시스템 요구사항 관리
```

```

┌────────────────┬──────────────┬──────────────┬──────────────┐
│                │ IEC 62304    │ ISO 26262-6  │ DO-178C      │

  구동계 SW: IEC 62304 Class C + ISO 26262 영향

  (ISO 13485: 품질 경영 시스템 추가 적용)
```

## Agile과 IEC 62304 (AAMI TIR45)

```
전통적 워터폴 vs Agile + IEC 62304:

AAMI TIR45:2012 권고 (Agile + IEC 62304 공존 방법):

  스프린트 기반 개발에서 IEC 62304 요구 충족:

  Sprint 계획:
    ├── User Story → SW 요구사항(SRS) 매핑
    ├── 위험 영향 분석 수행
    └── 테스트 케이스 작성 (TDD 방식)

  Sprint 구현:
    ├── 코드 작성 + MISRA 체크
    ├── 단위 테스트 + 커버리지 측정
    └── 코드 리뷰 (Class B/C 필수)

  Sprint 완료:
    ├── RTM 업데이트 (스프린트 항목 연결)
    ├── 변경 이력 기록 (Change Request 연결)
    └── 회귀 테스트 실행

  릴리즈:
    ├── 전체 IEC 62304 문서 패키지 완성
    ├── 독립 검증 (V&V) 수행
    └── 의료기기 허가 제출 패키지 준비

핵심 원칙:
  "민첩성을 위해 추적성을 포기하지 않는다"
  → 자동화 도구로 추적성 부담 최소화
  → Jira+GitHub 연동: 이슈 ID = SRS ID = TC ID
```
