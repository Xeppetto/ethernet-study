# ISO 26262 (Road Vehicles — Functional Safety)

## 개요
ISO 26262는 자동차의 전기/전자(E/E) 시스템의 우발적 오류로 인한 사고를 방지하기 위해 제정된 기능 안전(Functional Safety) 국제 표준입니다(2018년 2판, 12개 파트). **의도치 않은 오작동**에 의한 안전 위험을 허용 가능한 수준(ALARP)으로 낮추는 것이 핵심 목표입니다.

의료 로봇 분야에서는 IEC 62304(의료기기 SW), IEC 60601-1이 1차 적용 표준이지만, 수술 로봇의 고도 전자 제어 시스템에 ISO 26262 방법론이 점점 광범위하게 채용됩니다. 특히 ASIL 분류 체계, HARA, E2E 보호, FFI(Freedom from Interference) 개념은 의료 안전 표준과 직접 호환됩니다.

> **의료 로봇 관점**: 수술 로봇의 관절 제어 ECU는 ASIL D 수준의 안전 무결성이 요구됩니다. HARA를 통해 "관절 속도 제한 소프트웨어 오작동 → 환자 충격"과 같은 시나리오를 분석하고, E2E 보호 프로파일(AUTOSAR E2E Profile 4)로 Ethernet 제어 메시지의 무결성을 보장합니다. `IEC_62304.md`의 Class C 요구사항과 교차 적용하며, `HIL.md`의 HIL 테스트로 HARA에서 도출된 안전 목표를 검증합니다.

## ASIL (Automotive Safety Integrity Level)

### ASIL 등급 및 결정 공식

```
ASIL 등급 (A~D + QM):
  ASIL D ← 가장 엄격 (생명 위협 수준의 위험)
  ASIL C
  ASIL B
  ASIL A ← 최소 안전 요구사항
  QM    ← 품질 관리 수준 (안전 무관)

결정 공식: ASIL = f(Severity × Exposure × Controllability)

Severity (S) - 사고 심각도:
  S0: 부상 없음
  S1: 경상 (치료 가능, 완치)
  S2: 중상 (생명 위협이 아닌 심각한 부상)
  S3: 사망 또는 영구 장해 (생명 위협)

Exposure (E) - 위험 상황 노출 빈도:
  E0: 발생 불가능 (이론적 가능성)
  E1: 매우 드물게 (< 1% 운전 시간, 연 몇 회)
  E2: 드물게 (1~10%, 월 몇 회)
  E3: 보통 (10~50%, 주 몇 회)
  E4: 높음 (> 50%, 거의 항상)

Controllability (C) - 운전자 제어 가능성:
  C0: 일반적으로 제어 가능 (거의 모든 운전자)
  C1: 쉽게 제어 가능 (대부분 제어 가능)
  C2: 보통 수준 (일부 어려움)
  C3: 제어 어려움 또는 불가능 (< 90% 성공)
```

### ASIL 결정 매트릭스

| S / (E, C) | E1 C1 | E1 C2 | E1 C3 | E2 C1 | E2 C2 | E2 C3 | E3 C1 | E3 C2 | E3 C3 | E4 C1 | E4 C2 | E4 C3 |
|------------|-------|-------|-------|-------|-------|-------|-------|-------|-------|-------|-------|-------|
| **S1** | QM | QM | QM | QM | QM | A | QM | A | B | A | B | C |
| **S2** | QM | QM | A | QM | A | B | A | B | C | B | C | D |
| **S3** | QM | A | B | A | B | C | B | C | D | C | D | D |

## HARA (Hazard Analysis and Risk Assessment)

```
HARA 절차:

1단계: Operational Situation Analysis
  로봇 운용 상황 목록:
    OS-01: 수술 중 (환자 연결 상태)
    OS-02: 환자 이송/위치 조정
    OS-03: 기구 교체/청소
    OS-04: 소독/유지보수
    OS-05: 이동/위치 이동

2단계: Hazard Identification
  H1: 관절 속도 제어 실패 → 과속 동작
  H2: 그리퍼 힘 제한 실패 → 과도한 압력 인가
  H3: 위치 제어 오류 → 비정상 궤적
  H4: 통신 지연 → 명령 지연 실행
  H5: 비상정지 미작동 → 정지 불가

3단계: ASIL 결정 (S/E/C 조합)
  H1 (OS-01): S3, E4, C3 → ASIL D
  H2 (OS-01): S3, E3, C3 → ASIL D
  H3 (OS-01): S2, E3, C2 → ASIL B
  H4 (OS-01): S2, E3, C2 → ASIL B
  H5 (OS-01): S3, E3, C3 → ASIL D

4단계: Safety Goal 정의
  SG-01 [ASIL D]: 수술 중 로봇 관절 속도 제한 초과 없음
  SG-02 [ASIL D]: 그리퍼 힘은 어떤 상황에서도 5N을 초과하지 않음
  SG-03 [ASIL D]: 비상정지 명령 수신 후 100ms 이내 정지

HARA 결과 문서 구조:
┌─────┬────────┬─────────────┬───────────────┬────┬────┬────┬──────┬──────┐
│ ID  │ 상황   │ 위험원       │ 위험 이벤트     │ S  │ E  │ C  │ ASIL │ Goal │
├─────┼────────┼─────────────┼───────────────┼────┼────┼────┼──────┼──────┤
│ H1  │ 수술 중 │ 관절 과속    │ 환자 조직 손상  │ S3 │ E4 │ C3 │ D    │ SG-01│
│ H2  │ 수술 중 │ 힘 제한 실패  │ 환자 압박 부상  │ S3 │ E3 │ C3 │ D    │ SG-02│
│ H3  │ 수술 중 │ E-Stop 실패  │ 정지 불가       │ S3 │ E3 │ C3 │ D    │ SG-03│
│ H4  │ 이동 중 │ 충돌 감지 실패│ 보조 직원 충돌  │ S2 │ E3 │ C2 │ B    │ SG-04│
└─────┴────────┴─────────────┴───────────────┴────┴────┴────┴──────┴──────┘
```

### E2E (End-to-End) Protection 프로파일

```
AUTOSAR E2E Profile 비교:

Profile 1: 1 Byte Counter + 1 Byte CRC8
  → 소규모 신호 (< 16 Byte 페이로드)
  → CAN 기반 시스템과 하위 호환

Profile 2: 1 Byte Counter + 1 Byte CRC8 + Data ID
  → 주요 ECU 간 신호 (권장)
  → 현재 가장 많이 사용

Profile 4: 2 Byte Counter + 4 Byte CRC32
  → 대용량 Ethernet 데이터 (Safety-Critical 영상 등)

Profile 6: 16 Byte CRC 128 + 4 Byte Counter
  → 가장 강력 (ASIL D 최고 보안)

E2E Profile 2 헤더 구조 (4 Byte 추가):
 ┌──────────────┬─────────────┬──────────────────────────────┐
 │ Counter (1B) │  CRC (1B)   │       Payload (N Byte)       │
 └──────────────┴─────────────┴──────────────────────────────┘
  0~14 순환      CRC-8/SAE J1850

E2E 검증 상태:
  OK       : 모든 검사 통과 → 정상 동작
  REPEATED : Counter 반복 → 중복 무시
  LOST     : Counter 건너뜀 → 결함 처리
  CRC_ERR  : CRC 불일치   → 결함 처리 (안전 상태 전환)
  TIMEOUT  : 수신 타임아웃  → 결함 처리 (비상정지)
```

### E2E 구현 코드 (Python, 개념)

```python
import struct

class E2EProfile2:
    """AUTOSAR E2E Profile 2 구현 (개념 코드)"""
    CRC8_TABLE = [...]   # CRC-8/SAE J1850 룩업 테이블

    def __init__(self, data_id: int):
        self.data_id = data_id
        self.counter = 0

    def protect(self, payload: bytes) -> bytes:
        """E2E 헤더 추가"""
        cnt = self.counter & 0x0F
        crc = self._crc8(bytes([cnt,
                                (self.data_id >> 8) & 0xFF,
                                self.data_id & 0xFF]) + payload)
        self.counter = (self.counter + 1) % 15
        return bytes([cnt, crc]) + payload

    def check(self, data: bytes, last_counter: int) -> str:
        """E2E 수신 검증"""
        cnt, rcv_crc = data[0] & 0x0F, data[1]
        payload = data[2:]
        expected_crc = self._crc8(bytes([cnt,
                                         (self.data_id >> 8) & 0xFF,
                                         self.data_id & 0xFF]) + payload)
        if rcv_crc != expected_crc:
            return 'CRC_ERR'
        if cnt == last_counter:
            return 'REPEATED'
        if cnt != (last_counter + 1) % 15:
            return 'LOST'
        return 'OK'

    def _crc8(self, data: bytes) -> int:
        crc = 0xFF
        for b in data:
            crc = self.CRC8_TABLE[(crc ^ b) & 0xFF]
        return crc ^ 0xFF
```

## Freedom from Interference (FFI)

```
FFI 목적:
  ASIL D 기능이 QM 기능의 결함으로 인해 영향받지 않도록 보장

FFI 구현 방법:
  1. 공간 분리 (Spatial Separation)
     - MPU (Memory Protection Unit): 메모리 영역 격리
     - IOMMU: DMA 접근 제한
     예: ASIL D 파티션은 MPU로 보호된 전용 RAM 영역 사용

  2. 시간 분리 (Temporal Separation)
     - Time Partition (AUTOSAR OS): CPU 시간 슬롯 할당
     - Hypervisor (Xen, ACRN): 가상화 기반 분리
     예: ASIL D 태스크는 전체 CPU 시간의 30% 전용 할당

  3. 통신 분리
     - VLAN: 네트워크 트래픽 격리
     - 메시지 인증 (AUTOSAR SecOC)
     예: ASIL D ECU는 전용 VLAN + E2E로만 통신

ECU 파티션 구조 예시:
┌─────────────────────────────────────────────────────┐
│  Safety ECU (단일 SOC, Dual Core)                   │
│                                                     │
│  Core 0 (ASIL D 파티션):    Core 1 (QM 파티션):     │
│  ┌─────────────────┐        ┌──────────────────┐   │
│  │ - E-Stop 로직    │        │ - UI 처리         │   │
│  │ - 힘 모니터링    │        │ - 영상 처리       │   │
│  │ - 속도 제한      │        │ - 클라우드 통신   │   │
│  │ MPU 보호 영역    │        │ 일반 메모리       │   │
│  └─────────────────┘        └──────────────────┘   │
│  Time Partition 0           Time Partition 1        │
│  (30% CPU, 고정)             (70% CPU)              │
│                                                     │
│  QM 파티션 무한루프/오버런 → Core 0에 영향 없음 (FFI)  │
└─────────────────────────────────────────────────────┘
```

## 하드웨어 안전 메커니즘 (FMEDA)

```
FMEDA (Failure Modes, Effects, and Diagnostic Analysis):

목표 지표:
  PMHF (Probabilistic Metric for Random Hardware Failure):
    ASIL D: < 10⁻⁸ /h (10 FIT) ← 가장 엄격
    ASIL C: < 10⁻⁷ /h (100 FIT)
    ASIL B: < 10⁻⁷ /h
    FIT = 1 failure per 10⁹ hours

  DC (Diagnostic Coverage, 진단 커버리지):
    ASIL D: High DC > 99% (최고)
    ASIL C: Medium DC > 90% 이상
    ASIL B: Low DC > 60%

  SPF Metric (Single Point Fault):
    ASIL D: ≥ 99% (단일 고장점이 Safety Goal 침해 불가)
    ASIL C: ≥ 97%

FMEDA 분석 표 (예시, 토크 센서):
```

## ASIL 분해 (Part 9 - ASIL Decomposition)

```
ASIL 분해 원칙:
  단일 ASIL D 요구사항을 두 독립 채널로 분해

분해 규칙 (ISO 26262-9):
  D → C(D) + A(D)   (C와 A 합쳐서 D 수준)
  D → B(D) + B(D)
  C → B(C) + A(C)
  C → A(C) + QM(C)
  B → A(B) + QM(B)

의료 로봇 힘 제한 기능 분해 (ASIL D):
  SG: 수술 중 관절 토크 한계 초과 없음 [ASIL D]
              │
    ┌─────────┴──────────┐
    │                    │
채널 A [ASIL B(D)]    채널 B [ASIL B(D)]
  토크 센서 피드백 제어    독립 힘 모니터링
  속도 프로파일 제한       + 하드웨어 E-Stop 회로
  (SW 구현)               (독립 MCU + 릴레이)

두 채널이 독립적으로 동작 → 하나 실패해도 나머지가 보호
```
