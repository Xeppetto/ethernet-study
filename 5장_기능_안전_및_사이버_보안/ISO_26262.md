# ISO 26262 (Functional Safety)

## 개요
ISO 26262는 자동차의 전기/전자(E/E) 시스템의 오류로 인한 사고를 방지하기 위해 제정된 기능 안전(Functional Safety) 국제 표준입니다(2018년 2판). 의료 로봇 분야에서는 IEC 62304(의료기기 SW)와 IEC 60601-1이 주로 사용되지만, 수술 로봇의 고도 전자 제어 시스템에서 ISO 26262의 방법론이 점점 광범위하게 적용되고 있습니다.

**핵심 원칙**: 시스템이 의도치 않게 작동하거나 고장 났을 때 발생할 수 있는 위험을 **허용 가능한 수준(ALARP: As Low As Reasonably Practicable)**으로 낮추는 것입니다.

## ASIL (Automotive Safety Integrity Level)

```
위험도 분류 (ASIL A~D):

ASIL D ← 가장 엄격 (생명 위협 수준의 위험)
ASIL C
ASIL B
ASIL A ← 최소 안전 요구
QM    ← 품질 관리 수준 (안전 요구 없음)

ASIL 결정 공식:
  ASIL = f(Severity, Exposure, Controllability)

Severity (S):
  S0: 부상 없음
  S1: 경상 (치료 가능)
  S2: 중상 (생명 위협 아님)
  S3: 사망 또는 영구 장해

Exposure (E):
  E0: 발생 불가
  E1: 매우 드물게 (< 1% 운전 시간)
  E2: 드물게 (1~10%)
  E3: 보통 (10~50%)
  E4: 높음 (> 50% 운전 시간)

Controllability (C):
  C0: 일반적으로 제어 가능
  C1: 쉽게 제어 가능
  C2: 보통 수준 제어
  C3: 제어 어려움 또는 불가능
```

### ASIL 결정 매트릭스 예시

| S / E / C | C1 | C2 | C3 |
|-----------|----|----|-----|
| S1, E2    | QM | QM | QA |
| S2, E3    | QM | A  | B  |
| S3, E3    | A  | C  | D  |
| S3, E4    | B  | C  | D  |

## HARA (Hazard Analysis and Risk Assessment)

```
HARA 절차:

1단계: Situation Analysis (운전 상황 분석)
  예: 수술 중, 이동 중, 보정 중, 청소 중...

2단계: Hazard Identification (위험원 식별)
  예: 로봇 팔 의도치 않은 움직임, 과도한 힘 인가...

3단계: ASIL 결정
  각 Hazardous Event에 대해 S/E/C 평가 → ASIL

4단계: Safety Goal 정의
  예: "수술 중 로봇 팔은 의도치 않은 5N 이상의 힘을 인가해서는 안 된다."
  → Safety Goal에 ASIL 부여 (예: ASIL D)

HARA 결과 문서 구조:
  ┌─────────────────────────────────────────────────────────────────────┐
  │ ID  │ 상황    │ 위험원     │ 위험 이벤트  │ S │ E │ C │ ASIL │ Goal │
  ├─────┼────────┼──────────┼────────────┼──┼──┼──┼──────┼──────┤
  │ H1  │ 수술 중 │ 과속 동작  │ 환자 조직 손상│S3│E4│C3 │ASIL D│SG-01 │
  │ H2  │ 이동 중 │ 급정지 실패 │ 충돌 부상   │S2│E3│C2 │ASIL B│SG-02 │
  └─────────────────────────────────────────────────────────────────────┘
```

## Ethernet 통신과 기능 안전

### Black Channel 통신

```
Black Channel 개념:

┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  Safety ECU (Tx)         Black Channel         Safety ECU (Rx)  │
│  ┌──────────┐            (Ethernet)            ┌──────────┐     │
│  │ 제어 데이터│                                 │ 제어 데이터│     │
│  │ CRC 계산  ├──── [Switch] ── [Cable] ─────────►│ CRC 검증  │     │
│  │ Seq# 추가 │   (신뢰 불가, "Black Channel")   │ Seq# 확인 │     │
│  │ 타임스탬프 │                                 │ 타임아웃  │     │
│  └──────────┘                                 └──────────┘     │
│                                                                  │
│  Safety Layer (Application): E2E Protection                     │
│  Network Layer (Ethernet): 안전 기능 없음 (Black Channel)         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘

오류 탐지 메커니즘:
  - CRC/Checksum: 데이터 비트 오류 탐지
  - Sequence Number: 메시지 손실/중복 탐지
  - Timestamp: 시간 지연/타임아웃 탐지
  - Source ID: 잘못된 발신자 탐지
  - Data ID: 데이터 타입 혼동 방지
```

### E2E (End-to-End) Protection

```
AUTOSAR E2E Profile 예시 (Profile 2):

E2E 헤더 (최소 4 Byte 추가):
┌──────────┬──────────┬────────────┬──────────────────┐
│ Counter  │ CRC (1B) │ Data ID    │   실제 페이로드   │
│  (1B)    │          │ (High/Low) │                  │
└──────────┴──────────┴────────────┴──────────────────┘

E2E 검증 상태:
  OK       : 모든 검사 통과
  REPEATED : Sequence Number 반복 (메시지 중복)
  LOST     : Sequence Number 건너뜀 (메시지 손실)
  WRONG_SEQ: 잘못된 순서
  CRC_ERR  : CRC 불일치 (데이터 손상)
  TIMEOUT  : 수신 타임아웃

E2E 오류 시 Safety Mechanism:
  - 안전 상태(Safe State)로 전환
  - 비상 정지(Emergency Stop) 실행
  - 상위 시스템에 오류 보고
```

### Freedom from Interference (FFI)

```
ASIL 분리 (Independence) 요구사항:

┌─────────────────────────────────────────────────────┐
│               Single ECU (Partitioned)              │
│                                                     │
│  ┌─────────────────┐    ┌──────────────────────┐   │
│  │ ASIL D 파티션    │    │ QM 파티션             │   │
│  │ - 관절 안전 제어  │    │ - 영상 처리           │   │
│  │ - 힘 모니터링    │    │ - UI 처리            │   │
│  │ MPU 보호 영역    │    │ 일반 메모리 영역       │   │
│  └─────────────────┘    └──────────────────────┘   │
│           │                       │                 │
│      Time Partition 0         Time Partition 1      │
│      (전용 CPU 시간)           (나머지 시간)           │
│                                                     │
│  QM 파티션 오류 → ASIL D에 영향 없음 (FFI 보장)       │
└─────────────────────────────────────────────────────┘

FFI 기술:
  메모리: MPU (Memory Protection Unit), IOMMU
  시간:  Time Partition (AUTOSAR OS), Hypervisor
  통신:  VLAN 분리, 메시지 인증
```

## 하드웨어 안전 메커니즘

```
FMEDA (Failure Modes, Effects, and Diagnostic Analysis):

  목표 지표:
    PMHF (Probabilistic Metric for Random Hardware Failure)
      ASIL D: < 10⁻⁸ /h (10 FIT)
      ASIL C: < 10⁻⁷ /h (100 FIT)
      ASIL B: < 10⁻⁷ /h

    DC (Diagnostic Coverage): 진단 커버리지
      ASIL D: High DC > 99%
      ASIL C: High DC > 99% or Medium > 90%
      ASIL B: Low DC > 60%

    SPF (Single Point Fault): 단일 고장점
      ASIL D: SPF Metric ≥ 99%
      ASIL C: SPF Metric ≥ 97%

하드웨어 이중화 설계:
  ┌────────────┐    ┌────────────┐
  │ 채널 A (주) │    │ 채널 B (부) │
  │ - 센서 A1   │    │ - 센서 B1   │
  │ - MCU A    │    │ - MCU B    │
  └─────┬──────┘    └──────┬─────┘
        └──────── CRC ─────┘
              비교 모니터링
           (Voter / Comparator)
                   │
             ↓ 불일치 시
           Safety State
```

## ISO 26262 SW 요구사항 (Part 6)

```python
# ASIL D SW 코딩 지침 예시 (주석으로 규칙 표기)

# 규칙 1: 동적 메모리 할당 금지 (MISRA C Rule 20.4)
# 규칙 2: 재귀 함수 금지
# 규칙 3: 함수 포인터 사용 제한

# E2E 보호 구현 예시 (개념 코드)
import struct
import crc

class E2EProtection:
    """AUTOSAR E2E Profile 2 간략 구현"""

    DATA_ID = 0x1234  # 데이터 식별자 (설계 시 고정)

    def protect(self, payload: bytes, counter: int) -> bytes:
        """
        E2E 헤더 추가하여 전송 데이터 생성
        counter: 0~14 순환 (4비트)
        """
        # Data ID bytes
        data_id_high = (self.DATA_ID >> 8) & 0xFF
        data_id_low = self.DATA_ID & 0xFF

        # CRC 계산 (CRC-8 / SAE J1850)
        raw = bytes([counter, data_id_high, data_id_low]) + payload
        crc_val = crc.crc8(raw)

        return bytes([counter & 0x0F, crc_val]) + payload

    def check(self, data: bytes, last_counter: int) -> str:
        """
        E2E 검증
        반환값: 'OK', 'CRC_ERR', 'REPEATED', 'LOST', 'WRONG_SEQ'
        """
        counter = data[0] & 0x0F
        rcv_crc = data[1]
        payload = data[2:]

        # CRC 검증
        data_id_high = (self.DATA_ID >> 8) & 0xFF
        data_id_low = self.DATA_ID & 0xFF
        raw = bytes([counter, data_id_high, data_id_low]) + payload
        expected_crc = crc.crc8(raw)

        if rcv_crc != expected_crc:
            return 'CRC_ERR'

        # Sequence 검증
        expected_counter = (last_counter + 1) % 15
        if counter == last_counter:
            return 'REPEATED'
        elif counter != expected_counter:
            return 'LOST'

        return 'OK'
```

## ASIL 분해 (ASIL Decomposition)

```
ASIL D 요구사항을 두 독립 채널로 분해:

Safety Goal: ASIL D
          │
    ┌─────┴─────┐
    │           │
ASIL D       ASIL D
    │           │
    │  분해 적용  │
    ▼           ▼
ASIL B(D)   ASIL B(D)   ← 두 채널 합쳐서 ASIL D 달성
  채널 A      채널 B

분해 규칙:
  D → C + A (또는 A + C)
  D → B + B
  C → B + A (또는 A + B)
  C → A + QM (또는 QM + A)
  B → A + QM

의료 로봇 적용:
  힘 제한 기능 (ASIL D) →
    채널 A: 토크 센서 피드백 제어 (ASIL B)
    채널 B: 독립 힘 모니터링 + E-Stop (ASIL B)
```
